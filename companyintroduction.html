Technology Architecture/Topology



Why choose CEPH as the cornerstone of the fusion solution and Technology

1. GPFS
Gpfs (general parallel file system, is a distributed file system. Developed and owned by jbm, this is a proprietary, closed source storage system. This makes it unattractive and difficult to adapt to. Storage hardware plus authorization and support costs make it very expensive. In addition, it provides very limited access to storage; It can not provide block storage, nor can it provide restful access to the storage system, so it is a very restrictive system, and even the maximum number of data copies is limited to only three. This reduces the reliability of the system when multiple components fail at the same time
2. iRODS
Irdos is a representative of the current data system. It is an open source data management software released under the bsd protocol of 3-clause. Irdos is not a highly reliable storage system because its icat metadata server is a single point of (single point of failure.spof), and it does not provide a real ha. In addition. It provides limited storage interfaces; Can neither provide block storage nor provide restful interfaces to access the child storage system, so this is a very restrictive system that is more suitable for storing a small number of large files. Instead of storing small files and large files at the same time, irdos uses the traditional way to maintain an index of physical locations (associated with file names). Since multiple clients need to request file locations from the metadata server, the metadata server needs to be subjected to additional computational load, resulting in a single point of failure and performance bottlenecks
3. HDFS
Hdfs is a distributed extensible file system written in] ava and created for the hadoop framework. Hdfs is not a fully posix compatible file system and does not support block storage. This makes it less reliable than ceph.hdfs because it is not a highly available file system. The single point of failure and performance bottleneck in hdfs are mainly due to its single namenode node. It is more suitable for storing a small number of large files than for both small files and large files.
4. Lustre
Lustre is a parallel distributed file system driven by the open source community and released under the gnu (general public license, General Public license. In lustre, since there is only one single server responsible for storing and managing metadata. As a result, all I / O requests from the client depend entirely on the computing power of the server, but for enterprise computing, the server's computing power is usually low. Like irdos and hdfs, lustre is ideal for storing a small number of large files rather than both small files and large files. Like irdos, lustre manages an index file for mapping physical locations and filenames, which determines that it is a traditional architecture and prone to performance bottlenecks. Lustre does not have any node fault detection and correction mechanisms. In the event of a node failure, the client can only connect to other nodes on its own
5. Gluster
Glusterfs was originally developed by gluster, which was acquired by red hat in 2011. Glusterfs is a horizontally extended network attachment (network-attacbed) file system. In gluster, administrators must specify which placement strategy to use to store copies of data on different locales. G1uster itself does not have built-in block access, file systems and remote replicas, but is supported as an extension (add-ons).
6. CEPH
If we compare ceph with other existing storage solutions, because of the rich nature of ceph, it's obviously different. It overcomes the limitations of existing storage systems. And has proven to be an ideal replacement for expensive old storage systems that are open source software defined storage solutions running on any commercial hardware, This also makes it an economical storage solution. Ceph provides a variety of interfaces to enable clients to connect to ceph clusters, which adds flexibility to clients. For data protection, ceph does not depend on the frame of rald technology, because it has all the limitations mentioned above, but adopts replica and erasure code scheme which has been proved to be better than raid.
Each component of ceph is reliable and supports high availability. If you keep redundancy in mind when configuring ceph components, we can confidently say that ceph does not run at any single point of failure. And one of the biggest challenges of today's other storage solutions, ceph, is its unified nature, which provides off-the-shelf storage solutions for blocks, files, and objects. Other storage systems still do not have this feature. Ceph is suitable for storing both small and large files without any performance problems.
Ceph is a distributed storage system, the client can interact with ceph quickly it does not use the traditional data storage method to maintain a metadata bound to a physical location or file. Instead, it introduces a new mechanism that allows clients to dynamically compute the location of the data they need. This improves client performance. Because they no longer have to wait for the location and content of data to be retrieved from the metadata server. In addition, within the ceph cluster, the location of data is absolutely transparent and automated; clients and administrators do not need to worry about where data is stored in different areas of the radiation barrier, and ceph's intelligent systems manage all this automatically
